1. Goal & scope

Give Niyam a real identity + cloud-synced practice, with required Google Sign-In, without ever making the blocking engine depend on the network. Payment stays with Google Play. This spec covers:

  • Phase 2 — Foundation: Supabase SDK + client singleton + DB schema + RLS. No visible change yet.
  • Phase 3 — Required sign-in: native Google sign-in gate after Welcome; logout in Settings; session caching.
  • Phase 4 — Favourites sync and Phase 5 — streaks/journey sync + server-side entitlement check.

Founder decisions locked: sign-in gate sits after Welcome, before onboarding; on first sign-in we push local state up to the account, then sync (nothing lost).

2. Architecture

BackendSupabase — Auth (Google), Postgres (+ Row-Level Security), later Edge Functions for Play purchase verification (P5).

New package com.myniyam.app.backend: SupabaseClientProvider (lazy singleton), AuthRepository (sign-in/out, session state), SyncRepository (push/pull). The engine packages (service/, overlay/) are NOT touched.

3. Auth flow & sign-in placement

Placement (locked): Welcome → SignIn → onboarding (intention/mantra/language/apps) → permissions → Home. Returning users with a cached session skip straight past SignIn.

Sign-in sequence

1. SignInScreen shows "Continue with Google".
2. Credential Manager requests a Google ID token (serverClientId = Web client ID).
3. supabase.auth.signInWithIdToken(provider=Google, idToken, nonce).
4. On success: session persisted by supabase-kt (encrypted DataStore); ensure a profiles row exists; proceed to onboarding.
5. On cancel/failure: stay on SignInScreen with a retry; the user cannot proceed (sign-in is required).

Logout

Settings → new "Account" section showing the signed-in email + Sign out. Sign out clears the Supabase session and returns to SignInScreen. Local practice data is left on device (re-synced on next sign-in).

4. The offline rule (non-negotiable)

The blocking engine must never wait on the network. The AccessibilityService, overlay, grace timer, and streak-on-Continue all read/write local UserPrefs only. Sign-in is a one-time gate at first run; after that the cached session means the app opens straight to Home with no network call required. Sync is a background best-effort mirror. If Supabase is unreachable, nothing in the core loop breaks.

Engine-touch rule still applies: no changes to service/ or the overlay detection logic. Sync hooks live in repositories and ViewModels, not the engine.

5. Database schema + RLS

Four tables, all keyed by auth.uid(), all protected by RLS so a user can only read/write their own rows. SQL you'll run in the Supabase SQL editor (after you approve this spec — I won't run migrations for you):

-- profiles: one row per user
create table public.profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  email text,
  created_at timestamptz default now()
);

-- practice_state: the single synced practice snapshot (1 row per user)
create table public.practice_state (
  user_id uuid primary key references auth.users(id) on delete cascade,
  current_mantra_id text,
  sadhana_start_epoch_day bigint,
  completed_mantra_ids text[] default '{}',
  selected_intention text,
  display_language text,
  streak_count int default 0,
  updated_at timestamptz default now()
);

-- favourites: one row per (user, mantra)
create table public.favourites (
  user_id uuid references auth.users(id) on delete cascade,
  mantra_id text not null,
  created_at timestamptz default now(),
  primary key (user_id, mantra_id)
);

-- entitlements: server-trusted premium state (written by Edge Function in P5)
create table public.entitlements (
  user_id uuid primary key references auth.users(id) on delete cascade,
  premium_active boolean default false,
  premium_plan text,
  trial_start_epoch_day bigint,
  source text,            -- 'play' once server verification lands
  updated_at timestamptz default now()
);

alter table public.profiles       enable row level security;
alter table public.practice_state enable row level security;
alter table public.favourites     enable row level security;
alter table public.entitlements   enable row level security;

-- Each user sees and writes only their own rows.
create policy "own profile"  on public.profiles
  for all using (auth.uid() = id)       with check (auth.uid() = id);
create policy "own state"    on public.practice_state
  for all using (auth.uid() = user_id)  with check (auth.uid() = user_id);
create policy "own favs"     on public.favourites
  for all using (auth.uid() = user_id)  with check (auth.uid() = user_id);
-- entitlements: user may READ own; writes restricted to service role (Edge Function).
create policy "read own entitlements" on public.entitlements
  for select using (auth.uid() = user_id);
Reads/writes from the app use the anon key + the user's session JWT; RLS enforces per-user isolation. The service_role key (full bypass) lives only in Edge Functions on the server — never in the app.

6. What syncs, when, conflict handling

DataDirectionTrigger
practice_state (streak, journey, intention, language)two-waypush on change (debounced); pull on sign-in & app resume
favouritestwo-waypush on toggle; pull on sign-in
entitlementspull only (P5)app resume; server writes via Play webhook

First sign-in (locked: push-then-sync)

On the very first successful sign-in, the app uploads whatever is already on-device (streak, current journey, completed list, favourites) to the new account, then flips to normal two-way sync.

Conflict rule

Last-write-wins on updated_at, with a max() merge for monotonic fields (streak count, completed set is a union). Practical for a single-user-multi-device app; no fancy CRDT needed. Engine never blocks on a sync result.

10. Open questions for you

  1. Account deletion in-app: Play requires an in-app + web account-deletion path for apps with accounts. OK to add a "Delete account" action in Settings during P3 (calls an Edge Function)? (Recommended yes.)
  2. Sign-in failure with no internet at first run: since sign-in is required and needs network, a brand-new install with no connectivity can't get past SignIn. Acceptable? (Standard for account-required apps; the engine still can't run without setup anyway.)
  3. Trial/premium today is local sandbox. Until P5's server verification, entitlements stay local; the entitlements table is created now but written later. Fine?
