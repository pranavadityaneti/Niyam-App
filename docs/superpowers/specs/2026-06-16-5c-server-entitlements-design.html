1. Goal & why now

Today premium is decided only on the device: UserPrefs.premiumActive and the 7-day trial start live in local DataStore. Phase 7 made the purchase real, but the device is still the source of truth — which means two holes:

  • Trial-reinstall loophole: uninstall + reinstall resets the local trial start → unlimited free 7-day trials.
  • No cross-device premium: a paid user on a new phone has to tap "Restore" and Play must still have the purchase locally; nothing carries their entitlement by account.

5c makes the server the source of truth. A Supabase Edge Function verifies the Play purchase with Google's API and writes the trusted entitlements row (the table already exists from P2, RLS read-own, service-role write-only). The app trusts that row and reconciles on launch.

In plain English: the app stops believing the phone about who paid. After a purchase, our server asks Google "is this real?", records the answer under the user's account, and every device that signs in reads that record. Reinstalling no longer resets the trial, and a paid user gets premium back automatically on any device.

2. The one decision for you

How does our server learn about subscription changes (renewals, cancellations, refunds)?

OptionWhat it meansInfra
Verify-on-demand RECOMMENDED for now The app sends the purchase token to our function after purchase and on each launch; the function asks Google for current status and updates the row. Status can be stale between launches (fine for a blocker app). Just a Google service account. No Pub/Sub.
RTDN (Real-time Developer Notifications) LATER Google pushes renew/cancel/refund events to a Pub/Sub topic; a function consumes them and updates the row within seconds, even if the app never opens. Pub/Sub topic + push endpoint + Play Console wiring.
My recommendation: ship verify-on-demand in 5c (closes both holes with the least setup), and keep RTDN as a clearly-scoped follow-up once revenue justifies the extra plumbing. For a launch-stage app where users open it daily, on-launch reconciliation is enough — a cancelled user keeps premium until at most their next app open, which is acceptable.

  • Server-truth: entitlements row, written only by the function (service role). The app can read its own row but never write it (existing RLS).
  • Local cache: UserPrefs.premiumActive / trialStartEpochDay become a cache of the server row so the engine and paywall keep working offline.

4. The verify-entitlement Edge Function

Deno function at supabase/functions/verify-entitlement/index.ts. Same JWT pattern as delete-account: identify the caller from their token, never trust a user id sent in the body.

Input (POST, JSON): { productId, purchaseToken } — the values from the Play Purchase.

Steps:

  1. Resolve caller's user.id from the Authorization JWT (anon client).
  2. Authenticate to Google with the service-account key (stored as a Supabase secret) → get an access token for the Android Publisher API.
  3. Call purchases.subscriptionsv2.get with packageName=com.myniyam.app + purchaseToken.
  4. Read the subscription state: SUBSCRIPTION_STATE_ACTIVE / _IN_GRACE_PERIOD → entitled; _CANCELED but still within paid term → entitled until expiry; _EXPIRED/_ON_HOLD → not entitled.
  5. Map the line-item product id → plan name (niyam.premium.*).
  6. upsert the entitlements row with service role: premium_active, premium_plan, source='play', updated_at=now().
  7. Return { premium_active, premium_plan } so the client can mirror immediately.

The service-account JSON key is set once via supabase secrets set GOOGLE_PLAY_SA_KEY=... (or a file) — it is not the Supabase service_role key and never ships in the app.

5. Client behaviour

New backend/EntitlementSync.kt (a thin object, never on the engine path):

After a successful purchase

PlayBillingGateway.persistAndAcknowledge() additionally calls the function with the purchase's productId + purchaseToken; the returned trusted state is mirrored to UserPrefs. (If the call fails, the local grant from Phase 7 still stands — see failure handling.)

On launch (signed in)

  1. Read the user's entitlements row via Postgrest.
  2. If premium_active → ensure local premium is set (restores paid users on a new device with no Play action).
  3. If the row says not active but local thinks premium → clear local premium (server wins).
  4. Re-verify any locally-owned Play purchase (from queryPurchasesAsync) to refresh the row.

Trial moves to the server

When the trial starts (onboarding completion), write trial_start_epoch_day + source='trial' to the user's entitlements row (this write is the one exception — done client-side under RLS write-own? No: entitlements is service-role-write-only, so the trial start is written by a tiny function call too, or we relax the RLS to allow the user to set their own trial_start once). Decision in spec review: simplest is a start-trial path in the same function; I'll fold it in. On launch the client adopts the server's trial_start if present, so a reinstall can't reset it.

6. Trust model & failure handling

  • Server wins, but never harshly offline. Entitlement is only downgraded on a definitive server "not active" response — not on a network error. A failed verify/launch call leaves the existing local state untouched (a paying user on a flaky connection is never locked out mid-session).
  • Engine untouched. The blocking engine reads local UserPrefs only; all of 5c is in backend/ + billing glue. Zero service-file edits.
  • Idempotent. Re-verifying the same token just rewrites the same row.

7. Setup you'll need to do

YOUR SETUP
  1. Google Cloud service account in the project linked to Play Console; enable the Google Play Android Developer API.
  2. In Play Console → Users & permissions, grant that service account access to view financial data / manage orders & subscriptions (the scope needed to read subscription status).
  3. Download the service-account JSON key and hand it to me to set as a Supabase secret (I'll run supabase secrets set — or you do, I'll give the exact line). The key never enters the app or the repo.
  4. Deploy the function: supabase functions deploy verify-entitlement (same flow as delete-account).

All of this can wait until the closed-test build from Phase 7 exists, since 5c can only be tested with a real purchase.

9. Non-goals (this phase)

  • RTDN / Pub/Sub real-time updates — deferred (see §2).
  • Refund / chargeback automation beyond what an on-launch verify naturally reflects.
  • Proration, plan upgrades/downgrades UI — Play handles the billing; we only read entitlement.
  • Practice-state & favourites sync — that's Phase 5, the next step after 5c.
Decisions flagged for your review: (a) verify-on-demand now vs RTDN — I recommend on-demand; (b) trial-start written via the same function vs relaxing RLS — I recommend the function path (keeps entitlements service-role-write-only). Tell me if either should change and I'll revise before building.
