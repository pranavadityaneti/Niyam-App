1. Method & scope

Five reviewers ran in parallel, each owning one subsystem and reading its files in full: blocking engine + pause/interval, backend / sync / auth / billing, data / prefs / progress, UI / navigation / onboarding / settings, and Supabase Edge Functions + SQL. They were told to hunt only for real defects (crashes, races, logic errors, security holes, leaks, fail-open paths, edge cases) with file:line + severity — no style nits.

Verification gate: I did not accept any reviewer claim at face value. Every Critical/High was re-checked against the actual code before fixing — which is how two "criticals" were correctly downgraded to non-issues (§5). All fixes compile, pass 125/125 unit tests, and the engine was touched in exactly one place.

2. Critical — fixed (4)

CRITICAL · FIXED

Delete-account left all read-history in the device database

"Delete account" wiped preferences but never cleared the Room read-events table. That's a privacy leak, and because a fresh journey starts at day 0, the leftover history was counted — so a brand-new mantra could show an inflated or instantly-"complete" streak.
In plain English: deleting your account didn't actually erase your practice history from the phone, and the next person could inherit a fake streak. Now deletion fully wipes it.
Fix: ReadEventDao.clearAll + ProgressRepository.clearAll, awaited in the delete flow.
CRITICAL · FIXED

The new interval check-in barely fired

Showing the mantra overlay emits a window event for Niyam's own package, which was being treated as "user left the blocked app" and cancelled the interval timer — so the every-30/60/120-min check-in was repeatedly torn down instead of counting.
In plain English: the feature you just asked me to build was quietly defeating itself. Fixed — the overlay no longer stops its own timer.
Fix: exclude our own package from the interval-cancel path in AppLockAccessibilityService.
CRITICAL · FIXED

Launch reconcile could revoke a paying user's premium

A trial-only entitlements row carries premium_active=false by column default. The launch reconcile read that as "subscription inactive" — so a user who paid but whose purchase-verification hadn't yet reached the server could be downgraded to free on next open.
In plain English: a real subscriber could get bumped back to free by accident. Now premium is only ever revoked on a definitive Google "not active," never off a trial record.
Fix: reconcile grants on true; revokes only on an authoritative source="play" false.
CRITICAL · FIXED (needs deploy)

One subscription could unlock unlimited accounts

The Google Play purchase token wasn't bound to a buyer, so a genuine active token could be replayed from other accounts to grant them premium.
In plain English: someone could share or reuse one paid subscription across many accounts. Now the first account to redeem a token owns it; others are rejected.
Fix: verify-entitlement records the token + rejects cross-account reuse; migration 0003 adds the column + unique index. Requires your redeploy.

3. High — fixed (5)

HIGH · FIXED

Sign-in could clobber an established device's newer data

Seed-from-server overwrote local practice on every sign-in with no "newer wins" check. A sign-out/in cycle on an active device could replace fresh local choices with a staler server copy.
Fix: only seed onto a fresh, un-onboarded device.
HIGH · FIXED

Favourites sync could wipe everything on a partial failure

Push did delete-all-then-insert; a crash or network drop between the two left zero favourites on the server, which a later fresh-device seed would then read back as "no favourites."
Fix: diff-based add/remove — never a zero-rows window.
HIGH · FIXED

Auth gate could wrongly bounce a real user

The gate read a frozen onboarding snapshot (leaving the in-onboarding window unguarded) and didn't confirm there was no live session — so a transient "not authenticated" during a token refresh could kick an authenticated user to sign-in and wipe their back stack.
Fix: read onboarding state reactively + skip the bounce whenever a session still exists.
HIGH · FIXED (needs deploy)

Server could revoke premium on an ambiguous Play state

verify-entitlement wrote premium_active=false for any non-active state, including transient ones (on-hold, paused, pending). And it trusted the client-sent product id for the plan label.
Fix: revoke only on terminal states (expired/revoked/canceled-past-expiry); plan taken strictly from Google. Requires your redeploy.
HIGH · FIXED

Billing purchase could resolve the wrong flow or hang forever

A single shared result holder meant two overlapping purchases could cross-resolve, and if Play never called back the caller could hang indefinitely.
Fix: serialize purchases behind a Mutex + a bounded timeout.

4. Medium — fixed

MEDIUM · FIXED

Hardening batch

  • Empty OTA catalog now rejected → falls back to the bundled mantras instead of showing only the om fallback.
  • Unlock grace cleared when the block-list changes → no stale 5-minute free pass after unblock/re-add.
  • Home first-line guarded against empty mantra text (no crash on malformed remote content).
  • RemoteConfig free-set parse uses a safe cast → one bad config element can't abort the whole config apply.
  • (Server) Edge-function error responses no longer leak internal Google/Postgres detail; sync-trial stops clobbering the source column and bounds the trial day.

5. Verified & dismissed (not real)

DISMISSED

"Interval check-in fires during the 5-minute grace"

Can't happen — the interval is at minimum 30 minutes, always well past the 5-minute grace window.
DISMISSED

"Continue button can get stuck disabled on a config change"

The overlay is a WindowManager view, not an Activity — it isn't recreated on configuration changes, so the countdown reliably finishes and enables Continue.

6. Real but deferred

DEFERRED · DO BEFORE REAL BILLING

Acknowledge-retry on launch

If a purchase acknowledgement fails, Play auto-refunds after 3 days. The durable fix is to re-acknowledge any owned-but-unacknowledged purchase at app start. Not exercised today (no live products), so safe to do right before enabling real purchases.
DEFERRED · MINOR

Lower-impact polish

  • Accessibility consent isn't re-recorded if the service was already enabled when the user reaches that step (Play disclosure edge case).
  • Trial reminder can be missed if the daily worker lands exactly on the day-6→7 boundary.
  • Progress layer uses wall-clock dates (clock travel could nudge streak/day-N).
  • In-memory preference snapshot does read-modify-write without a lock (self-heals on restart).
  • Announcement dedupe keys on the title (a blank-title announcement can't be dismissed permanently).
  • Settings "blocked apps" count can read stale after returning from the editor.
  • The pause onboarding step keeps its state outside the onboarding view-model (back-then-forward loses an unsaved edit).
