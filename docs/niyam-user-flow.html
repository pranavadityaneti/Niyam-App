1. Install & first launch

Technical
User installs the APK/AAB (min SDK 26 / Android 8, target SDK 35). On first launch MainActivity.onCreate runs: UserPrefs.ensureLoaded (DataStore → in-memory snapshot), RemoteConfig.ensureLoaded (cached OTA config, if any) + an async RemoteConfig.refresh, theme applied, and a force-update gate check (this build's versionCode vs server min_supported_version_code). Start destination = WELCOME because onboardingComplete == false. Chrome locale defaults to English (displayLanguage = ENGLISH) via LocaleBridge in attachBaseContext.
In plain English
The user downloads Niyam and opens it. It starts in English and shows a Welcome screen with the brand lockup and the line "A pause before the scroll." Nothing about their phone or other apps has been touched yet. They tap Get started.

2. Sign-in (required)

Technical
Welcome → SIGN_IN. SignInScreen uses Google Sign-In via Credential Manager (ComposeAuth.googleNativeLogin with the Web client ID) → Supabase Auth session. A profiles row is auto-created server-side (DB trigger) on first sign-in. On success, PracticeSync.seedFromServerIfPresent runs: if the account already has server practice (returning user on a fresh device) it is pulled into local prefs, onboarding is marked complete, and the app recreates straight to Home; otherwise the new user proceeds to onboarding. Login is required — a reactive gate in AppNavHost bounces any signed-out user on a post-onboarding screen back to Sign-in.
In plain English
The user signs in with their Google account — this is mandatory, so their practice can be saved and synced. We only receive their email; we never see their password. If they're a returning user on a new phone, their mantras, progress and favourites are restored automatically and they land straight on the Home screen. A brand-new user continues to set up the app.
Why required: sign-in is what makes cross-device sync, the server-verified subscription, and "delete my data" possible — and it closes the loophole where someone reinstalls to restart the free trial.

3. Onboarding

Technical
Four steps, each persisted to UserPrefs via OnboardingViewModel, with a back arrow on steps 2–4:
  1. Language → displayLanguage (English, Hindi, Sanskrit-Devanagari, Telugu, Tamil, Kannada, Bengali, Gujarati). Applied app-wide on completion via activity recreate + LocaleBridge.
  2. Intention → selectedIntention (Focus / Calm / Sadhana / Dharma / Devotion).
  3. Mantra → currentMantraId (the starting sadhana mantra).
  4. Apps → blockedPackages (chosen from AppCatalog.APPS, which is OTA-overridable; the engine will block any chosen package, the catalog is just the shortlist).
In plain English
The user picks, in order: the language they want the app and mantras in; their intention (why they're here — focus, calm, daily practice, dharma, or devotion); their starting mantra; and the apps they want to block (Instagram, YouTube, etc., plus a few games). After picking the language, the whole app switches to it.

4. Permissions

Technical
Four permission screens, in order: Overlay (SYSTEM_ALERT_WINDOW, to draw the mantra over other apps) → Accessibility (BIND_ACCESSIBILITY_SERVICE, the launch-detection engine — gated behind a Play-compliant prominent-disclosure card + explicit "I agree — turn on" consent, timestamp persisted) → Battery (REQUEST_IGNORE_BATTERY_OPTIMIZATIONS, so OEMs don't kill the service) → OEM autostart (per-manufacturer walkthrough; "Done" primary on stock Android). The roll-up PermissionChecker.allPermissionsGranted checks overlay + accessibility + battery.
In plain English
The app asks for the permissions it needs to do its one job — show a mantra the moment a blocked app opens. The most important one (Accessibility) comes with a clear, upfront explanation of exactly what it does and doesn't do (it only notices which app you opened — never reads your screen or messages), and the user must explicitly agree before it's turned on. The battery and autostart steps make sure the phone doesn't silently shut Niyam down.

5. Trial starts (onboarding complete)

Technical
When the OEM step finishes, UserPrefs.setOnboardingComplete + UserPrefs.startTrial(today) fire, then the activity recreates → Home in the chosen locale. trialStartEpochDay is set; Entitlements.TRIAL_DAYS = 7. The trial start is also pushed to the server (sync-trial, earliest-wins) so a reinstall can't restart it. Entitlements.state(premiumActive, trialStartEpochDay, today) now returns TRIAL.
In plain English
The moment setup is done, a 7-day free trial of Premium starts automatically — there's no "before the trial" waiting period. The user lands on the Home screen with full access to everything. The trial's start date is remembered on our servers, so uninstalling and reinstalling won't give them a fresh trial.

6. The core daily loop

Technical
A foreground service (specialUse) hosts AppLockAccessibilityService. On TYPE_WINDOW_STATE_CHANGED, it reads the foreground package name only; if it's in blockedPackages and not within the 5-minute per-package UnlockGrace, OverlayManager draws the mantra overlay (TYPE_APPLICATION_OVERLAY) within ~200 ms. The overlay shows the current mantra (script + transliteration + meaning in the chosen language) with a 15-second countdown ring. Tapping Continue dismisses it, records a read (Room read_events → streak/day-N math), and grants a 5-minute grace for that package. Opening a non-blocked app hides the overlay (OverlayHideDecision).
In plain English
This is the heart of the app. When the user opens an app they chose to block, Niyam instantly appears first with their mantra and a 15-second timer ring. They read the mantra, tap Continue, and the app opens normally. For the next 5 minutes that app won't interrupt them again — so navigating around inside it isn't annoying. The mantra never blocks them forever; it's a pause, not a wall. Each Continue counts toward their daily streak and their "Day N of 14" sadhana progress.

Display languagesAll 9English & Hindi onlyAll 9

Favourites, streak, sync✓✓✓
Technical
Gating is computed by Entitlements. isPremiumExperience(state) is true for TRIAL and PREMIUM. Free set = Entitlements.activeFreeMantraIds (default 5: gita-2-47, mahamrityunjaya, gayatri, gita-4-7-8, hanuman-chalisa-opening — one per intention; OTA-overridable). Free languages = English + Hindi. Grandfather rules: the user's current sadhana mantra and current language always stay usable even if they'd otherwise be premium-locked, so a trial user is never stranded mid-practice when the trial ends. Ads render only in FREE state and only if the ads_enabled flag is on.
In plain English
During the trial they get everything. After the trial, the free version keeps the core blocking feature fully working — that's never paywalled — but limits them to 5 mantras and 2 languages, and shows small banner ads on the Home and Library screens. Premium removes the ads and unlocks all 26 mantras and all 9 languages. Importantly, whatever mantra and language they were already using stays available even after the trial — we never yank away their active practice.
Honest note: the original brief imagined the free 5 being "your chosen intention's 5 mantras." The shipped build uses a fixed set of 5 (one strong mantra per intention) — the dynamic, intention-locked version is a queued enhancement, not yet built. Functionally the free user still gets 5 good mantras; they're just not auto-tailored to their intention yet.

8. During the 7-day trial

Technical
Entitlements.trialDaysLeft drives the Home/Settings status and the paywall timeline. A WorkManager daily job (TrialReminderWorker → TrialReminder logic) posts one notification around day 6 ("your trial ends tomorrow"), guarded so it fires at most once (trialReminderShown). Clock-rollback is guarded (a backdated device clock can't extend the trial).
In plain English
Through the week the app quietly shows how many trial days are left. On day 6 the user gets a single, gentle notification that the trial ends tomorrow, with a tap-through to subscribe if they want to keep Premium. No nagging, no repeated pop-ups.

9. After the trial (free tier)

Technical
On the 8th day, Entitlements.state returns FREE (exclusive 7-day boundary). Library shows lock icons on the 21 premium mantras; tapping a locked mantra routes to the paywall. The language editor locks all but English/Hindi (marked "· Premium"). Ad banners begin rendering on Home + Library. The user keeps their streak, favourites, current mantra, and current language (grandfathered).
In plain English
When the trial ends, the app keeps doing its main job — blocking apps with a mantra — for free, forever. The user now sees small ads, and the extra mantras and languages show a lock. If they tap a locked mantra, they're taken to the subscribe screen. Nothing they were already using disappears.

10. Subscribing to Premium

Technical
PaywallScreen offers Weekly / Monthly / Yearly plans (display prices ₹15 / ₹49 / ₹399; the real charge comes from Google Play Console). Purchase goes through Billing.gateway → PlayBillingGateway (Play Billing Library 7): connect → query product → launch Play's purchase sheet → on success persist premium locally + acknowledge the purchase → call the verify-entitlement Edge Function, which verifies the token with Google's Play Developer API and writes the server-trusted entitlements row. Restore re-grants from any owned subscription. On every launch, EntitlementSync.reconcileOnLaunch mirrors the server entitlement into local state (so Premium follows the account across devices, and a cancellation eventually revokes locally).
In plain English
The user picks a plan and pays through Google Play — we never handle card details. Once paid, ads vanish and everything unlocks. Their Premium is tied to their account, so it works on any device they sign in to, and "Restore purchase" brings it back if they reinstall. Our server double-checks every purchase with Google so the subscription can't be faked.
Setup status: the billing code is complete, but real purchases need the 3 subscription products created & active in Play Console + an active payments profile. Until then, debug builds use a sandbox (fake) purchase for testing.

11. Cross-device, sign-out, delete account

Technical
Sync: PracticeSync pushes practice + favourites to Supabase on app stop; on a fresh device, sign-in seeds them back. Sign out (Settings → Account): clears the session, returns to Sign-in (practice stays on device). Delete account (Settings → Account): confirmation → delete-account Edge Function deletes the auth user with the service-role key (DB tables cascade) → local UserPrefs.clearAll → returns to Welcome as a fresh device.
In plain English
Practice syncs automatically, so a new phone picks up where the old one left off. From Settings → Account the user can sign out (their stuff stays on the phone, ready to sync again) or permanently delete their account — which erases everything we hold on our servers and resets the app on the device. There's also a public myniyam.com/delete-account page explaining how, as Google requires.

12. Updates without a new build (OTA)

Technical
RemoteConfig + ContentSync read a public app_config table + a Storage content/mantras.json at launch (offline-first; build values are the fallback). Changeable without a build: the free mantra set, the blockable-app catalog, paywall trust copy, feature flags (e.g. ads_enabled), the force-update minimum version + message, a Home announcement banner, and the entire mantra catalog (edit JSON, bump content_version). Not changeable OTA (Play rule): app code, screens, the engine.
In plain English
We can change a lot of the app's content and settings without shipping a new version to the Play Store — add or fix a mantra, change which apps are offered, turn ads off, post an announcement, or force everyone onto a new version in an emergency. Anything that's actual app code (new features, the blocking engine) still needs a normal Play update — that's Google's rule, not a choice.

Honest notes & edge cases

  • No "before the trial" gap. The trial begins automatically at the end of onboarding — the user never sits in a pre-trial limited state.
  • Free tier is a fixed 5 mantras, not yet auto-matched to the user's intention (queued enhancement).
  • Offline: the engine, mantras, and overlay all work with no internet after the first sign-in. Sync, entitlement re-checks, and OTA simply resume when a connection returns; nothing blocks the core loop.
  • The mantra is never a hard wall — every block can be passed in 15 seconds. Niyam is a mindful pause, not a lock.
  • Privacy: the accessibility service reads only the foreground package name, on-device; it never reads screen content, messages, or other apps' data, and that package name never leaves the phone.
